# Step 02 Design - Multi-Horizon Dataset

## Purpose

The current `SmoothPredictor` receives `horizonMs` as an input feature, but it predicts only one displacement for the currently sampled horizon. v24 needs to test whether a model trained with labels over several future horizons behaves better when the user changes target correction.

## Dataset Shape

Each training row should contain one runtime-safe feature vector and several labels:

```text
features(t) -> label(t + h0), label(t + h1), ..., label(t + hk)
```

The feature vector must be computed only from the current and past cursor samples. Future/reference cursor samples are allowed only for labels and metrics.

## Horizon Grid

The initial grid follows the product target-correction range and 60 Hz timing:

- base refresh: 16.6667 ms
- target correction display values: -32, -24, -16, -8, 0, 8, 16, 24, 32 ms
- extra future probes: 0, 4, 8, 12, 16, 24, 32 ms

The dataset builder should store the actual requested horizon in milliseconds for every label. Negative or already-expired horizons should be kept as diagnostic rows but excluded from normal training unless a candidate explicitly handles them as hold/current-position labels.

## Required Columns

Runtime input columns:

- package id, split, quality bucket, duration bucket, scenario id;
- sample timestamp, target timestamp, refresh period, stopwatch frequency;
- current cursor position;
- history-derived velocity windows for 2, 3, 5, 8, and 12 samples;
- recent segment maximum speed;
- latest delta;
- path net/path/efficiency;
- runtime target displacement estimate;
- direction vector;
- missing-history indicators or equivalent history availability counts.

Label columns:

- horizon id;
- horizon milliseconds;
- future target `dx/dy`;
- future lead/lag direction information for metrics;
- validity flag for interpolation coverage.

Regime labels for evaluation only:

- static;
- slow/medium/fast;
- abrupt stop;
- resume after stop;
- high scheduler delay or high-load capture;
- normal 60 Hz capture.

## Splitting

Use the v21 split manifest as the first source of truth. Do not randomly mix rows from the same scenario across train and test.

Preferred split strategy:

- train: existing train packages and generated training-only MotionLab packages;
- validation: existing validation package plus a held-out generated package;
- test: existing normal-quality test package and new 60 Hz validation packages;
- robustness: high-load, poll-delayed, abrupt-stop, and scheduler-delay packages.

## Metrics

Every score must be reported both overall and by horizon bucket:

- visual p95/p99;
- lead p99;
- lag p95/p99;
- stop-event peak lead max;
- return motion max;
- stationary jitter p95;
- invalid/expired horizon rate.

## Notes

The first dataset builder can be C# because the v21 harness already contains most parsing and interpolation logic. Python is acceptable for training-side export if it reads compact JSON/CSV generated by the C# builder. Large raw row dumps should stay untracked.
